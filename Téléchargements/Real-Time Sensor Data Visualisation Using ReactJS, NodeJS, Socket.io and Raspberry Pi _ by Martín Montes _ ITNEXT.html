Real-Time Sensor Data Visualisation Using ReactJS, NodeJS, Socket.io and Raspberry Pi

In this article, we’ll have a look at a feature that I’ve implemented in an Internet of Things platform. It consists of displaying a real-time chart of the measurements performed in the registered things of the platform.

Architecture

For this case, a “thing” is a Raspberry Pi with temperature and humidity sensors that exposes its data using sockets through its IP.

Those things are registered in a NodeJS backend which will store its information in a database and publish its measurements in a MQTT broker. Also, the backend will give the possibility of connecting to a thing via sockets, which is the case we will see in further detail.

Finally, there is a ReactJS frontend that will be in charge of requesting to connect to a thing and render its data in a real time chart.

Wiring things up

Let’s start by setting up the hardware. For this use case we will need:

  • Raspberry Pi 3
  • DHT11: Temperature and humidity sensor
  • DS18B20: Waterproof temperature sensor. We will be using it for the outdoor temperature.
  • 4 x 220Ω Resistor
  • 3 x LED (Green = OK, Red = KO, Yellow = Operation in progress)

Hardware abstractions @ Thing

Abstraction is a CS concept that consists in hiding the details of an entity to focus in solving a problem. In our case, the entities would be the hardware components and the problem to solve would be getting the sensor’s data and give feedback to the user by turning on the LEDs.

We will also need a few more classes to orchestrate these hardware components. SensorHandler and LEDHandler will do that job.

Socket.io server @ Thing

As previously mentioned, things will expose its data through a socket. For doing so, we will run a Socket.io server in them.

The main idea behind Socket.io is asynchronously communicating two machines by using an API based on events that use websockets underneath. SocketHandler will take care of emitting those events in a configured interval of time. Note that their content will be the result of calling SensorHandler.read():

Socket.io server @ Backend

The backend will have a Socket.io server as well that will act as a middleware between the client and the thing. To achieve that, we will configure a middleware in our Socket.io instance that will have the following responsibilities:

  • Check authentication by validating a JWT.
  • Validate and get the thing name that the client has requested to connect to.
  • Validate and get the measurement type that the client has requested.

If everything is fine, the next step will be actually opening a socket to the thing and forward the data received back to the client. This process is done in the SocketController class which is quite similar to the thing’s class SocketHandler .

Wrapping Up

We’ve built a fullstack JavaScript solution that allows the user to visualise real time information about his/her things. Maybe a cool feature could be triggering actions, for example, by implementing a worker that subscribes to a MQTT topic and sends a message using a Telegram bot if the value obtained reaches a threshold. Guess what ? It is already done and available on GitHub. Another article is coming… probably.

Also, an interesting reafactoring could be replacing Socket.io with gRPC, which also allows us to have a bidirectional streaming of data. Guess what? No, this is not implemented… yet.
